Containers don't provide security isolation by themselves — a misconfigured container running as root with host access is no safer than a process on the host. Securing containers requires intentional configuration at the image, runtime, and orchestration layers.

## Dockerfile Hardening

### Run as Non-Root

By default, Docker containers run as root inside the container. If the container is compromised, the attacker has root access to the container namespace and potentially the host.

```dockerfile
# ✗ Default — runs as root
FROM node:20

# ✓ Switch to a non-root user
FROM node:20

RUN groupadd -r appgroup && useradd -r -g appgroup appuser

WORKDIR /app
COPY --chown=appuser:appgroup . .
RUN npm ci --only=production

USER appuser  # all subsequent commands and the container process run as this user

EXPOSE 3000
CMD ["node", "server.js"]
```

### Minimize the Attack Surface

```dockerfile
# ✓ Use distroless or minimal base images
FROM gcr.io/distroless/nodejs20-debian12

# Distroless images contain only the runtime — no shell, no package manager,
# no debugging tools. An attacker who gets RCE has almost nothing to work with.

# ✓ Or use Alpine for a small, auditable image
FROM node:20-alpine

# ✓ Multi-stage builds — build dependencies never reach the final image
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
CMD ["node", "dist/server.js"]
```

### Pin Image Digests

Image tags are mutable — `node:20` today and next month may be different images. Pin by digest for reproducibility:

```dockerfile
# ✗ Mutable tag
FROM node:20-alpine

# ✓ Immutable digest
FROM node:20-alpine@sha256:a1b2c3d4...  # exact image locked
```

### No Secrets in Layers

Docker layers are cached and stored in the image — any secret written during build is accessible in the image history.

```dockerfile
# ✗ Secret baked into layer
RUN pip install --extra-index-url https://user:password@private.registry.com/simple/ mypackage

# ✓ Use BuildKit secret mount (not baked into image)
# syntax=docker/dockerfile:1
RUN --mount=type=secret,id=pip_conf,dst=/root/.config/pip/pip.conf \
    pip install mypackage
```

## Runtime Security Controls

### Read-Only Root Filesystem

Prevent the running process from modifying the filesystem — most application code should not need to write anywhere except designated data paths:

```yaml
# docker-compose.yml
services:
  app:
    image: myapp:latest
    read_only: true
    tmpfs:
      - /tmp:size=100m  # writable tmpfs for temp files if needed
    volumes:
      - uploads:/app/uploads  # writable persistent volume
```

### Drop Capabilities

Linux capabilities are fine-grained root privileges. Containers inherit a default set — drop everything not needed:

```yaml
# docker-compose.yml
services:
  app:
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE  # only if binding to port <1024
```

### Seccomp and AppArmor

System call filtering restricts what system calls a container process can make:

```bash
# Docker Desktop ships with a default seccomp profile
# For Kubernetes, use a security context:
```

```yaml
# Kubernetes Pod SecurityContext
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault  # applies the container runtime's default seccomp profile
  containers:
    - name: app
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        runAsNonRoot: true
        runAsUser: 1000
        capabilities:
          drop: ["ALL"]
```

## Image Scanning

Scan images for known CVEs before deployment:

```bash
# Trivy — fast, comprehensive scanner
trivy image myapp:latest

# Grype — alternative scanner
grype myapp:latest

# Snyk
snyk container test myapp:latest
```

Integrate into CI:

```yaml
# GitHub Actions
- name: Scan image for vulnerabilities
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: 'myapp:${{ github.sha }}'
    format: 'sarif'
    exit-code: '1'           # fail build on HIGH/CRITICAL
    severity: 'HIGH,CRITICAL'
```

## Kubernetes Security

### Pod Security Standards

Kubernetes defines three built-in security levels via Pod Security Admission:

| Level | Policy | Use for |
|---|---|---|
| `privileged` | No restrictions | System-level pods only |
| `baseline` | Prevents known privilege escalations | General workloads |
| `restricted` | Hardened; non-root, no privilege escalation | Security-sensitive workloads |

```yaml
# Apply to a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

### Network Policies

By default, all pods in a Kubernetes cluster can communicate with each other. Network Policies limit this to explicitly allowed paths.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - port: 3000
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: database
      ports:
        - port: 5432
    - to:   # allow DNS
        - namespaceSelector: {}
      ports:
        - port: 53
          protocol: UDP
```

### RBAC for Service Accounts

Each pod runs with a service account. Limit what that service account can do:

```yaml
# Create a minimal service account
apiVersion: v1
kind: ServiceAccount
metadata:
  name: api-sa
  namespace: production
---
# Grant only what's needed
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: api-role
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["app-config"]  # only this specific secret
    verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: api-role-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: api-sa
roleRef:
  kind: Role
  name: api-role
  apiGroup: rbac.authorization.k8s.io
```

## Supply Chain: Image Signing

Sign container images so consumers can verify they were built by your pipeline and not tampered with:

```bash
# Cosign (Sigstore) — keyless signing via OIDC
cosign sign --yes myregistry.io/myapp:latest

# Verify
cosign verify --certificate-identity-regexp=".*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  myregistry.io/myapp:latest
```

## Security Checklist

- [ ] Containers run as non-root user
- [ ] Base images are minimal (distroless or alpine)
- [ ] Images pinned by digest in production
- [ ] No secrets in Dockerfile or image layers
- [ ] Images scanned for CVEs in CI pipeline
- [ ] Root filesystem set to read-only where possible
- [ ] All Linux capabilities dropped except those needed
- [ ] Kubernetes Network Policies restrict pod-to-pod traffic
- [ ] Pod Security Standards set to `restricted` for production namespaces
- [ ] Service accounts follow least-privilege RBAC
- [ ] Images signed and signature verified at deploy time
