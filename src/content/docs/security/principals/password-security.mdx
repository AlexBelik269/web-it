Passwords are the most attacked credential type in existence — and the most poorly managed. The majority of breaches involve stolen, weak, or reused credentials. Understanding password security means understanding both how attackers exploit passwords and how to make credential compromise survivable.

## How Passwords Are Attacked

<Callout type="note"> 
### 1. Brute Force
</Callout>
The attacker systematically tries every possible password until the correct one is found. For a 6-character lowercase password (26⁶ = 308 million combinations), a modern GPU can crack it in under 1 second against an MD5 hash.

Brute force is generally only practical against:
- Offline hash databases (no rate limiting)
- Weak or short passwords
- Fast hash functions (MD5, SHA-256 — as opposed to bcrypt/Argon2id)
<Callout type="note"> 
### 2. Dictionary Attacks
</Callout>
Instead of trying every combination, dictionary attacks use wordlists — lists of common words, phrases, known passwords, and variations. The **rockyou.txt** wordlist (from the 2009 RockYou breach) contains 14 million real passwords and is the starting point for virtually every cracking session.

```
rockyou.txt top 20 passwords:
123456, 12345, 123456789, password, iloveyou, princess, 1234567,
rockyou, 12345678, abc123, nicole, daniel, babygirl, monkey,
lovely, jessica, 654321, michael, ashley, qwerty
```

<Callout type="note"> 
### 3. Rule-Based Attacks
</Callout>
Tools like Hashcat apply transformation rules to wordlist entries:

| Rule | Example |
|------|---------|
| Capitalize first letter | `password` → `Password` |
| Append numbers | `password` → `password123`, `password1` |
| Substitute common chars | `password` → `p@$$w0rd` |
| Append years | `password` → `password2023`, `password2024` |
| Append common suffixes | `password` → `password!`, `password.` |
| Reverse | `password` → `drowssap` |

A 10-character password that looks complex — `P@$$word1!` — is cracked in minutes because it follows predictable patterns from the wordlist + rules.
<Callout type="note"> 
### 4. Credential Stuffing
</Callout>
After a data breach, billions of username/password pairs are published on the dark web. Attackers take these lists and try every combination against other services — because **65% of people reuse passwords across multiple sites** (Google/Harris Poll, 2019).

If your email and password were exposed in a breach of a gaming forum, attackers will immediately try that combination on Gmail, PayPal, bank websites, and corporate portals.

**Tools like Sentry MBA, OpenBullet, and SilverBullet** automate credential stuffing at massive scale — testing millions of combinations per hour.

<Callout type="note"> 
### 5. Password Spraying
</Callout>
The inverse of brute force. Instead of trying many passwords against one account (which triggers lockout), password spraying tries one common password (`Winter2024!`) against thousands of accounts.

Particularly effective against:
- Corporate Active Directory environments with account lockout policies
- Services with weak lockout controls
- Situations where attackers want to maintain access without triggering alerts
<Callout type="note"> 
### 6. Phishing for Credentials
</Callout>
The most reliable way to steal a password: ask for it. A convincing phishing site that looks identical to the real login page will collect credentials directly — no cracking required.

**Real-time phishing proxies** (Evilginx2, Modlishka) act as a man-in-the-middle proxy to the legitimate site, capturing both credentials and the session token — bypassing SMS-based 2FA.

## Password Strength

### What Actually Makes a Password Strong

Strength is measured by the number of guesses required to crack it, which depends on:
1. **Length** — the biggest factor; each additional character multiplies the search space
2. **Character set** — using uppercase, lowercase, numbers, symbols increases the multiplier per character
3. **Unpredictability** — avoiding dictionary words, patterns, and personal information

| Password | Character set | Length | Guesses needed |
|----------|--------------|--------|---------------|
| `dog` | lowercase | 3 | 17,576 |
| `Dog1!` | mixed | 5 | 916 million |
| `correct horse battery staple` | words + spaces | 28 | 10⁵¹ (using word dictionary) |
| `Tr0ub4dor&3` | mixed | 11 | ~10²⁸ (but common pattern) |
| Random 16 characters | full ASCII | 16 | 10³² |

**XKCD 936 insight:** Four random common words concatenated (`correct horse battery staple`) is both more memorable and more secure than a shorter complex password (`Tr0ub4dor&3`) — because length dominates entropy calculation.

### NIST Password Guidelines (SP 800-63B, 2017 update)

The US National Institute of Standards and Technology revised its password recommendations significantly:

**Previous guidance (pre-2017):**
- Require complexity (uppercase, numbers, symbols)
- Force regular rotation (every 90 days)
- Use security questions

**Current NIST guidance:**
- **Focus on length**, not complexity — minimum 8 characters, support up to 64
- **Check against breached password databases** — reject passwords that appear in known breach data
- **No mandatory rotation** — rotate only when there is evidence of compromise
- **No complexity rules** — they lead to predictable patterns (Password1!)
- **No security questions** — they are easily researched and provide false security
- **Allow all printable ASCII** and Unicode (spaces, emoji, etc.)

### HaveIBeenPwned

The **HaveIBeenPwned** (HIBP) service, run by security researcher Troy Hunt, maintains a database of 12+ billion exposed account records from hundreds of breaches. You can check if your email or password appears in known breaches.

HIBP also provides a **Pwned Passwords API** that allows services to check new passwords against 700+ million known compromised passwords during registration — without exposing the actual password (uses k-anonymity: only the first 5 characters of the SHA-1 hash are sent to the API).

import hashlib
import requests

def is_password_pwned(password: str) -> int:
    sha1 = hashlib.sha1(password.encode()).hexdigest().upper()
    prefix, suffix = sha1[:5], sha1[5:]
    
    response = requests.get(f"https://api.pwnedpasswords.com/range/{prefix}")
    hashes = response.text.splitlines()
    
    for line in hashes:
        hash_suffix, count = line.split(':')
        if hash_suffix == suffix:
            return int(count)  # Password seen this many times in breaches
    return 0  # Not found in breaches

count = is_password_pwned("password123")
if count:
    print(f"This password has been seen {count:,} times in breaches.")

## Password Managers

A **password manager** generates, stores, and auto-fills unique, strong passwords for every site and service. This solves the core problem: humans cannot memorize dozens of unique strong passwords, so they reuse weak ones.

### How Password Managers Work

1. A **master password** is used to derive an encryption key (via Argon2id or similar KDF)
2. All stored passwords are encrypted with that key
3. The vault (encrypted blob) is stored locally or synced to the cloud
4. The master password never leaves the user's device — the service cannot see your passwords

**Zero-knowledge architecture:** The password manager service stores only your encrypted vault. Even if breached, they have no access to your actual passwords.

### What to Look For

| Feature | Why it matters |
|---------|---------------|
| Zero-knowledge architecture | Breach of the service doesn't expose your passwords |
| Strong encryption (AES-256 + Argon2id master password) | Protects the vault at rest |
| Open-source (auditable) | Can verify security claims independently |
| Browser extension with autofill | Prevents typosquatting phishing (extension matches exact domain) |
| Cross-platform sync | Practical for daily use |
| Emergency access | Recovery if you lose your master password |
| Breach monitoring | Alerts when stored sites appear in breaches |

**Recommended password managers:** 1Password, Bitwarden (open source), KeePass (offline, open source), Dashlane.

**The LastPass 2022 breach:** LastPass was breached in 2022; attackers exfiltrated encrypted password vaults. Users with weak master passwords or older vault iterations were at risk. This illustrates why master password strength is critical — and why open-source options with auditable security are preferable.

### Password Manager Usage Tips

- **Master password:** Long passphrase (20+ characters), unique, never used anywhere else, memorized (not written down except in a secure physical location)
- **One password per site:** Password manager generates a random 20+ character password for every account
- **Enable 2FA on the password manager itself:** Protects the vault if the master password is exposed
- **Audit saved passwords:** Most managers flag reused, weak, or breached passwords

## Multi-Factor Authentication (MFA)

MFA requires a second proof of identity in addition to a password. Even if a password is compromised, MFA prevents account takeover — the attacker also needs the second factor.

### MFA Factor Types

| Factor | Examples | Phishing resistant? |
|--------|----------|---------------------|
| **Something you know** | Password, PIN | No |
| **Something you have** | TOTP app, hardware key, phone | TOTP: No; Hardware key: Yes |
| **Something you are** | Fingerprint, face recognition | Platform-dependent |

### TOTP (Time-Based One-Time Passwords)

TOTP (RFC 6238) generates a 6-digit code that changes every 30 seconds, based on a shared secret and the current timestamp:

```
Code = HMAC-SHA1(secret, floor(time / 30)) truncated to 6 digits
```

Used by Google Authenticator, Authy, Microsoft Authenticator. Significantly better than no MFA — but vulnerable to real-time phishing proxies that capture both the password and the OTP and use them immediately.

### SMS 2FA

A one-time code sent via text message. The weakest form of 2FA:
- **SIM swapping:** Attackers social-engineer mobile carriers into transferring the victim's number to attacker-controlled SIM
- **SS7 attacks:** Telecom protocol vulnerabilities allow interception of SMS in some jurisdictions
- Better than nothing — stops automated credential stuffing — but should not be used for high-value accounts

### Hardware Security Keys (FIDO2/WebAuthn)

Hardware keys (YubiKey, Google Titan Key) use public-key cryptography:
1. During registration, the key generates a key pair for each site; the public key is stored by the site
2. During login, the site sends a challenge; the key signs it with the private key (which never leaves the key)
3. The key also cryptographically binds the origin — it will only respond to the exact registered domain, making phishing impossible

**Fully phishing-resistant:** Even if you are tricked into visiting a fake login page, the key will not sign the challenge because the origin doesn't match. This is why FIDO2 is the gold standard for high-security authentication.

## Passkeys: The End of Passwords

**Passkeys** are the consumer-facing implementation of FIDO2/WebAuthn. Instead of a password, login uses a device-based cryptographic key pair tied to biometric authentication (Face ID, fingerprint).

### How Passkeys Work

1. During registration, the device generates a key pair; the public key is sent to the site
2. During login, the site sends a challenge; the device authenticates the user (biometric), then signs the challenge with the private key
3. The signed challenge is sent back; the site verifies with the stored public key

**Advantages over passwords:**
- **No secret to steal** — the private key never leaves the device; servers store only public keys
- **Phishing-resistant** — cryptographically bound to the exact domain
- **No password reuse** — unique key pair per site
- **No server-side password database to breach** — no passwords to hash or protect

**Sync passkeys:** Apple, Google, and Microsoft allow passkeys to sync across devices within their ecosystem (iCloud Keychain, Google Password Manager, Windows Hello) — so losing a device doesn't lock you out.

### Adoption Status (2025)

Major platforms supporting passkeys: Google, Apple, Microsoft, Amazon, GitHub, PayPal, WhatsApp, LinkedIn, eBay. Adoption is accelerating; passkeys are expected to replace passwords for most consumer use cases within the next few years.

## Password Policy Best Practices

| Policy | Recommendation | Rationale |
|--------|---------------|-----------|
| Minimum length | 12+ characters | Length dominates entropy |
| Maximum length | 64+ characters | Support passphrases |
| Complexity requirements | Optional; do not mandate | Leads to predictable patterns |
| Rotation | Only on compromise evidence | Frequent rotation leads to pattern-based weak passwords |
| Breached password check | Required | NIST SP 800-63B |
| Account lockout | After 10 attempts, or progressive delays | Stops brute force; avoid hard lockout (DoS risk) |
| MFA | Required for all accounts | Required for high-value accounts; strongly recommended universally |
| Password reuse | Prevent last 10 passwords | Stops trivial rotation |
