## Why the Same-Origin Policy Exists

Before CORS, the browser had one blunt rule: a script from `https://bank.com` must never
silently read data from `https://mail.com`. Without this, any malicious page you visit
could use your browser — and your logged-in session cookies — as a weapon, making
authenticated requests to other sites and leaking the responses back to an attacker.

The critical insight is that **the browser is the enforcer, not the server.** The request
still goes out over the wire. The browser simply refuses to hand the response body to
JavaScript unless the origins match (or the server explicitly opts in via CORS headers).

**An origin is:** `scheme + host + port` — all three must match.

| URL | Same origin as `https://example.com`? |
|---|---|
| `https://example.com/other-page` | ✓ Yes (only path differs) |
| `http://example.com` | ✗ No (different scheme) |
| `https://api.example.com` | ✗ No (different subdomain) |
| `https://example.com:8443` | ✗ No (different port) |

**What SOP restricts:**
- JS reading responses from cross-origin `fetch` / `XMLHttpRequest`
- JS reading content inside a cross-origin `<iframe>`
- JS accessing `localStorage` or cookies belonging to a different origin

**What SOP does NOT restrict:**
- The request being *sent* — CORS only gates whether JS can *read* the response
- `<img>`, `<script>`, `<link>`, `<form>` — these predate SOP and have separate rules

## How CORS Works

CORS (Cross-Origin Resource Sharing) is the opt-in mechanism. When a server wants to
allow cross-origin JavaScript reads, it includes `Access-Control-Allow-Origin` in its
response. The browser checks this header before exposing the response to JS.

### Simple vs. Preflight Requests

The browser treats two categories differently:

**Simple requests** — GET, or POST with a standard content type like `text/plain` or
`application/x-www-form-urlencoded` — are sent immediately. The browser checks the
response headers after the fact and either allows or blocks the JS from reading the result.

**Non-simple requests** — PUT, DELETE, custom headers, or a JSON body — first trigger an
`OPTIONS` preflight request. The browser asks the server: "Are you willing to accept this
kind of request from this origin?" Only if the server approves does the actual request go
out. This protects older servers that might not be CORS-aware from receiving unexpected
mutation requests via browser JS.

```
# Preflight exchange for DELETE with a custom header

Browser → OPTIONS /resource
          Origin: https://app.example.com
          Access-Control-Request-Method: DELETE
          Access-Control-Request-Headers: Authorization

Server  → 204 No Content
          Access-Control-Allow-Origin: https://app.example.com
          Access-Control-Allow-Methods: GET, POST, PUT, DELETE
          Access-Control-Allow-Headers: Authorization, Content-Type
          Access-Control-Max-Age: 86400   ← cache this for 24 h

Browser: preflight approved → sends actual DELETE
Server  → 200 OK
          Access-Control-Allow-Origin: https://app.example.com

Browser: JS can now read the response ✓
```

`Access-Control-Max-Age` caches the preflight result so the browser skips the OPTIONS
roundtrip on subsequent requests to the same endpoint.

### Credentialed Requests (Cookies)

By default, cross-origin requests carry no credentials. To include cookies, both sides
must opt in — and wildcards are forbidden.

The client must request credentials explicitly:

```javascript
fetch('https://api.example.com/data', {
  credentials: 'include',
});
```

And the server must echo the exact origin (never `*`) alongside `Allow-Credentials`:

```
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
```

Using `credentials: 'include'` with `Access-Control-Allow-Origin: *` is explicitly
prohibited by the CORS spec. The browser will refuse it.

## Secure CORS Configuration

The safest pattern is an **allowlist with echo**: maintain a set of trusted origins, check
the incoming `Origin` header against it, and if it matches, reflect that exact origin back.
Always include `Vary: Origin` so CDNs and browser caches don't serve a stale
`Access-Control-Allow-Origin` header from one origin to a completely different origin.

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```javascript
// express-cors-middleware.js
const ALLOWED_ORIGINS = new Set([
  'https://app.example.com',
  'https://admin.example.com',
]);

function corsMiddleware(req, res, next) {
  const origin = req.headers.origin;

  if (origin && ALLOWED_ORIGINS.has(origin)) {
    // Echo back the matched origin — never just set '*'
    res.setHeader('Access-Control-Allow-Origin', origin);
    // Required: tells caches this response varies per origin
    res.setHeader('Vary', 'Origin');
  }

  // Handle the OPTIONS preflight before anything else touches the request
  if (req.method === 'OPTIONS') {
    if (origin && ALLOWED_ORIGINS.has(origin)) {
      res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
      res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');
      res.setHeader('Access-Control-Max-Age', '86400');
    }
    return res.sendStatus(204);
  }

  next();
}

module.exports = { corsMiddleware };
```

<TabItem label="Python">
```python
# Flask.py
from flask import Flask, request, make_response
from typing import Optional

app = Flask(__name__)

ALLOWED_ORIGINS = {
    "https://app.example.com",
    "https://admin.example.com",
}

def apply_cors_headers(response, origin: Optional[str]):
    """Echo back the origin only if it's on the allowlist."""
    if origin and origin in ALLOWED_ORIGINS:
        response.headers["Access-Control-Allow-Origin"] = origin
        # Vary: Origin tells caches that this response varies by origin,
        # preventing a cached Allow header for origin A being served to origin B.
        response.headers["Vary"] = "Origin"
    return response

@app.after_request
def cors_middleware(response):
    origin = request.headers.get("Origin")
    return apply_cors_headers(response, origin)

@app.route("/api/data", methods=["OPTIONS"])
def preflight():
    """Handle the OPTIONS preflight explicitly."""
    origin = request.headers.get("Origin")
    response = make_response("", 204)
    if origin and origin in ALLOWED_ORIGINS:
        response.headers["Access-Control-Allow-Origin"] = origin
        response.headers["Access-Control-Allow-Methods"] = "GET, POST, PUT, DELETE"
        response.headers["Access-Control-Allow-Headers"] = "Content-Type, Authorization"
        response.headers["Access-Control-Max-Age"] = "86400"
        response.headers["Vary"] = "Origin"
    return response

@app.route("/api/data", methods=["GET"])
def get_data():
    return {"message": "Hello from the API"}
```

```csharp
// Program.cs
using Microsoft.AspNetCore.Builder;
using Microsoft.Extensions.DependencyInjection;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddCors(options =>
{
    // Define a named policy. Naming it lets you apply it selectively
    // to specific controllers or globally as shown below.
    options.AddPolicy("AllowedOrigins", policy =>
    {
        policy
            // SetIsOriginAllowed gives you full control over origin validation.
            // A simple lambda checking a HashSet is safer than WithOrigins()
            // because it avoids subtle substring matching edge cases.
            .SetIsOriginAllowed(origin =>
            {
                var allowed = new HashSet<string>(StringComparer.OrdinalIgnoreCase)
                {
                    "https://app.example.com",
                    "https://admin.example.com"
                };
                return allowed.Contains(origin);
            })
            .AllowAnyMethod()
            .WithHeaders("Content-Type", "Authorization")
            .AllowCredentials(); // Requires a non-wildcard origin — handled above
    });
});

builder.Services.AddControllers();

var app = builder.Build();

// UseCors must come before UseRouting / UseAuthorization
app.UseCors("AllowedOrigins");
app.UseRouting();
app.UseAuthorization();
app.MapControllers();

app.Run();
```

```java
// CorsConfig.java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.cors.CorsConfiguration;
import org.springframework.web.cors.UrlBasedCorsConfigurationSource;
import org.springframework.web.filter.CorsFilter;

import java.util.List;

@Configuration
public class CorsConfig {

    private static final List<String> ALLOWED_ORIGINS = List.of(
        "https://app.example.com",
        "https://admin.example.com"
    );

    @Bean
    public CorsFilter corsFilter() {
        CorsConfiguration config = new CorsConfiguration();

        // setAllowedOrigins performs exact-match validation — safe.
        // Never use setAllowedOriginPatterns("*") with allowCredentials(true).
        config.setAllowedOrigins(ALLOWED_ORIGINS);
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
        config.setAllowedHeaders(List.of("Content-Type", "Authorization"));
        config.setAllowCredentials(true);   // requires non-wildcard origins
        config.setMaxAge(86400L);           // cache preflight for 24 h

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        // Apply to all routes; narrow to "/api/**" if needed
        source.registerCorsConfiguration("/**", config);

        return new CorsFilter(source);
    }
}
```

## Dangerous Patterns to Avoid

Each of these looks reasonable at first glance but introduces real vulnerabilities.

### Reflecting the origin without validation

Any origin in the world can read your API responses if you echo back whatever header
arrives without first checking it against an allowlist.

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```javascript
// DANGEROUS
const origin = req.headers.origin;
res.setHeader('Access-Control-Allow-Origin', origin); // no allowlist check!
```
</TabItem>
<TabItem label="Python">
```python
# DANGEROUS
origin = request.headers.get("Origin")
response.headers["Access-Control-Allow-Origin"] = origin  # no allowlist check!
```
</TabItem>
<TabItem label="C#">
```csharp
// DANGEROUS
var origin = Request.Headers["Origin"].ToString();
Response.Headers["Access-Control-Allow-Origin"] = origin; // no validation
```
</TabItem>
<TabItem label="Java">
```java
// DANGEROUS

response.setHeader("Access-Control-Allow-Origin", origin); // reflects blindly
```

### Bypass-prone regex matching

The following regex intends to allow only `example.com` subdomains but accepts
`https://evil-example.com` because `$` matches end-of-string, not end-of-origin.
Use exact-string lookup in a Set/HashSet instead — it's faster and immune to this.

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```javascript
// DANGEROUS
const trusted = /example\.com$/;
if (trusted.test(origin)) { /* grants access to evil-example.com */ }
```
</TabItem>
<TabItem label="Python">
```python
# DANGEROUS
import re
trusted = re.compile(r"example\.com$")
if trusted.search(origin):   # .search matches anywhere in the string
    pass
# Bypassed by: https://evil-example.com  ← ends with "example.com"
#              https://example.com.evil.com ← also matches
```
</TabItem>
<TabItem label="C#">
```csharp
// DANGEROUS
if (origin != null && origin.EndsWith("example.com"))
{
    // Bypassed by https://evil-example.com
}
```
</TabItem>
<TabItem label="Java">
```java
// DANGEROUS
if (origin != null && origin.endsWith("example.com")) {
    // Bypassed by https://evil-example.com
}
```
</TabItem>
</Tabs>

### Wildcard with credentials

Pairing `*` with `Allow-Credentials: true` is prohibited by the CORS spec. Browsers
reject it, and modern frameworks throw at startup to prevent the misconfiguration.

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```javascript
// DANGEROUS — browsers reject this combination
res.setHeader('Access-Control-Allow-Origin', '*');
res.setHeader('Access-Control-Allow-Credentials', 'true');
```
</TabItem>
<TabItem label="Python">
```python
# DANGEROUS — browsers will reject this combination

response.headers["Access-Control-Allow-Credentials"] = "true"
```
</TabItem>
<TabItem label="C#">
```csharp
// DANGEROUS — ASP.NET Core throws at startup to prevent this

```
</TabItem>
<TabItem label="Java">
```java

```
</TabItem>

### Trusting the `null` origin

Sandboxed iframes and local `file://` documents send `Origin: null`. Attackers can
deliberately generate this header using a sandboxed iframe — never treat it as trusted.

<Tabs syncKey="lang">

```javascript
// DANGEROUS

  res.setHeader('Access-Control-Allow-Origin', 'null');
}
```


## The `Vary: Origin` Header

When your CORS logic echoes back different `Access-Control-Allow-Origin` values
depending on the request, caches need to know the response is not the same for all
callers. Without `Vary: Origin`, a CDN or browser cache might serve a response
cached for `app.example.com` — with that origin in the header — to a request from
a different origin that isn't on your allowlist.

```javascript
res.setHeader('Vary', 'Origin');
```

## CORS Is Not a Complete Security Defense

Understanding what CORS protects against — and what it doesn't — prevents a dangerous
false sense of security.

**CORS does prevent:**
- Cross-origin JavaScript reading API responses. This is the primary scenario CORS
  addresses: a compromised third-party script trying to exfiltrate your API data using
  a victim's session.

**CORS does NOT prevent:**

- **CSRF**: Forms and `<img>` tags bypass CORS entirely. A malicious page can submit a
  `<form>` POST to your server using the user's cookies without any CORS negotiation
  happening. Use CSRF tokens or `SameSite=Strict` cookies for this threat.
- **Server-side damage**: The request is still sent and processed before the browser
  blocks the JS from reading the response. A state-changing endpoint hit by a simple
  POST is already done its work.
- **Same-origin attacks**: Scripts loaded from the same origin have full unrestricted
  access — CORS plays no role at all.

## Debugging Reference

| Symptom | Likely cause | Fix |
|---|---|---|
| "No 'Access-Control-Allow-Origin' header" | Server didn't return the header | Check your allowlist logic; verify the incoming `Origin` matches exactly |
| CORS works in dev, fails in prod | Different origin values between environments | Add the production domain to your allowlist |
| Preflight returns 404 or 405 | Server doesn't handle `OPTIONS` | Add an OPTIONS route or middleware that responds before other handlers |
| Cookie not sent cross-origin | Missing credentials config | Client needs `credentials: 'include'`; server needs `Allow-Credentials: true` and a non-wildcard origin |
| CDN serving wrong `Allow-Origin` header | Missing `Vary: Origin` | Add `Vary: Origin` to all CORS responses |
| `InvalidCorsConfigurationException` (Spring) | Wildcard + credentials | Use explicit origins instead of `"*"` |
| ASP.NET throws at startup | `AllowAnyOrigin().AllowCredentials()` | Use `SetIsOriginAllowed` with a real validation function |
